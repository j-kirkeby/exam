Accessibility

Summary

For accessibility we have followed the guidelines from Web Content Accessibility Guidelines (WCAG2) and used an automated test, the Web Accessibility Evaluation Tool (WAVE test), to find potential problems in the HTML code.

Sections

  • Automatic testing - WAVE
  • Coding for accessibility

Automatic testing - WAVE

The test revealed issues with the HTML code. Here is what we found:

  • Skipped from <h2> to <h4> headings
  • Redundant links: logo and “Home” in the header both links to index
  • Suspicious link text: links just named “here”, unclear what is linked to
  • Very low contrast: The footer has black background. The blue (default) and purple (visited) default color for links did not contrast well.

Try the test here: Webaim

Final results and changes

In the final results we have only one alert, which is for "Redundant links". We decided to keep this as the functionality of the logo linking back to the home page is common, and many users expect this feature. We don’t see any negative impact from leaving this unchanged.

We fixed the other issues:

  • Changed <h*> to make sure no levels are skipped
  • Made link text explanatory
  • Updated the color of links for the footer to be a lighter shade of blue/purple (kept the colors as people are used to this functionality).

Coding for accessibility

Web Content Accessibility Guidelines

We followed the Web Content Accessibility Guidelines (WCAG2) to make sure the page is functional and accessible. Here is what we did:

  • Provide text alternative to non-text content
    • We added alt-attribute to all images
  • Make it easier for users to see and hear content / make text readable and understandable
    • We used a readable font (Tahoma is considered one of the most accessible fonts)
    • We maintained a sufficient contrast between background and text
    • We scaled the content for to be usable on smaller screens
  • Help users navigate and find content
    • We added links in multiple places to help navigation. Header, footer and in text (front page).
  • Make content appear and operate in predictable ways
    • We followed a usual structure for the layout of the web-page and tagging of text

Scalability - Screen/Phone

We used the Mobile Simulator extension in Chrome to check that the page would be readable on smaller devices.

In the code we used @media queries to change layout and scaling for mobile. Mainly changing the grid layout to a single column for screen <700px wide.

Web Accessibility Initiative

The Web Accessibility Initiative have created the Web Content Accessibility Guidelines (WCAG2). Which provide insight for important aspects consider with regards to accessibility:

  • Provide text alternatives for non-text content
  • Make it easier for users to see and hear content
  • Make all functionality available from a keyboard
  • Help users navigate and find content
  • Make text readable and understandable
  • Make content appear and operate in predictable ways

A good user interface is...

  • Secure
  • Functional
  • Accessible
  • User friendly
  • Appealing

The bold aspects are most important for this course.
